This document attempts to describe and explain the process of creating Hypervisor documents, both "white papers" and API documents will be discussed. This document is a work in progress and will evolve as the process develops.

1. Introduction

The Hypervisor group has chosen DocBook as the documentation model. This means that our white papers are written in using DocBook "tags" and are currently in "SGML" files but can be easily migrated to "XML" since the same the DocBook Definition is being used. The process does take some learning, but we are hoping it will be reasonably easy to enable all the features that this model provides and at the same time make authoring as simple as possible.

2. Getting Started

Writing and rendering a document using DocBook requires the use of SGML/XML tags to mark regions of text and give them some meaning, as well as a set of tools to render or build and display the final output.

2.1. Resource Reference

There are several resources for DocBook that should be useful. In particular you should be armed with at least the DocBook Element Reference manual which is part of the DocBook: The Definitive Guide . You may also want to consider this tutorial on getting started by Mark Galassi.

2.2. Required Tools

The tools necessary to author, build and render documents are as follows:

  • A text editor such as: Emacs, XEmacs, vi, or even notepad.

  • A Diagramming package that can translate to EPS for PostScript® printing and JPEG or PNG for online publications. Currently we support both XFig/TransFig, and Dia version 0.88.1 or greater.

  • A make program that drives the building process. Though almost any make program will do, Hypervisor officially supports only GNU Make.

  • An implementation of DSSSL, a style language to format SGML or XML documents. There are two popular packages that perform this task: OpenJade, and James Clark's Jade . We currently use OpenJade version 1.3 or greater but they seem to be interchangeable.

  • If PostScript output is required then both a LaTex package that is Web2c based such as teTex, and the jadeTex package are required.

Other tools, though not required, can make authoring a white paper that much easier. The use of the nsgmls command can check your document for errors without actually rendering it.

If your editor of choice is Emacs or XEmacs, then you will find that proper installation of the PSGML , version 1.2.2 or later, to be invaluable.

2.2.1. For Red Hat® Linux

If you have a Red Hat Linux distribution then you can grab the following RPM packages and install them on your machine:

stylesheets-0.13rh-4
openjade-1.3-6
jadetex-2.7-4
tetex-1.0.6-10 and friends

Note: All packages except for stylesheets-0.13rh-4 are installed by default.

These packages can be easily downloaded from Rpmfind website.

3. Building the White Papers

All documents can be built in this directory by using the make command. The following targets are supported:

  • all or default: build all known documents rendering only HTML.

  • html: build all HTML rederings of the documents.

  • ps: build only the PostScript output of the DOCSRCS files.

  • filename.[html|rtf|mif|ps|pdf]: build only that single document.

  • filename: build only the HTML output of the document that filename.sgml represents.

4. Adding a New White Paper

Adding a new white paper to this directory is rather simple in comparison to authoring the actual paper. Just do the following:

  1. Copy another similar document you wish to create, for example, use this file ReadME.sgml to create a whitepaper.sgml.

  2. Edit file to taste.

  3. Build document periodically by entering make whitepaper. This will build both HTML and PostScript.

  4. View whitepaper.html with your favorite web browser or print or view, with GhostView, whitepaper.ps.

  5. Repeat steps 2 to 4 until finished.
